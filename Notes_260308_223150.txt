第一部分，当前项目中存在的问题
第二部分，我们总结的 Feature Toggle 七条使用规则
第三部分，项目中的真实案例以及改进措施
第四部分是一个简单总结

第一部分：Problem
首先是 当前项目中存在的一些问题。
在 zzzx 项目中，Feature Toggle 被广泛用于：
控制功能上线
控制功能对特定国家的启用
等等
但是随着项目长期演进，Toggle 的使用也逐渐产生了一些管理问题。
目前在 DCPCA 项目中：

Toggle 表中一共有一千五百多条条 Toggle 数据
其中有一些 Toggle 已经在开启状态（Y）下持续了超过 3年，一些toggle已经很难分辨其最初的设计目的，甚至toggle数据正在摧毁我们的项目性能，

这些问题带来的影响包括：
难以理解的代码逻辑
维护困难
降低app的性能。
一些 Toggle 实际上已经演变成了tech debt。

目前，这些问题 一部分已经完成治理，一部分仍然在持续改进中。为了更好地管理 Feature Toggle，我们在项目中总结出了 PowerApps 中 Feature Toggle 的七条使用规则。

好，我们来看有那些规则

规则一：Safe by Default（默认安全）
Feature Toggle 的默认状态应该是 关闭（OFF）。
只有当 Toggle 值明确为 Y 时，功能才应该被启用。
其他所有情况都应该被视为 功能未开启。
这样做可以避免一些问题：
1.未完成的功能不会被意外暴露给用户。
2.未完成的功能永远是隐藏状态
3.版本发布和release officer永远安全，不会因为在relaese当中，因为忘记将某个toggle设置为关闭，带来了不必要的麻烦

简单来说就是：
只有明确开启，功能才生效。

规则二：One Feature, One Toggle（一个功能一个 Toggle）
一个功能只应该对应 一个 Toggle。
不应该为同一个功能创建多个 Toggle。
如果需要控制生效的国家的范围，可以使用 Country List 来控制，而不是创建多个 Toggle。
例如：
ToggleName
CountryList
Feature_NewUI
US, CA, JP
这里：
Toggle 控制 功能是否开启
CountryList 控制 在哪些地区发布
这样可以保持 Toggle 结构简单。

规则三：Toggles Are Not Business Logic（Toggle 不是业务逻辑）
Feature Toggle 只应该回答一个问题：
这个功能是否开启？
不要把 Toggle 和复杂业务逻辑混在一起。
例如下面这种写法是不推荐的：
复制代码

If ToggleA AND Country = "US" AND Role = "Admin"
Toggle 的判断应该保持 简单清晰。
复杂逻辑应该放在业务逻辑层，而不是 Toggle 中。

规则四：Load Once, Use Everywhere（加载一次，到处使用）
Feature Toggle 应该在 应用启动时加载一次。
如果每次使用 Toggle 都查询数据源，会带来两个问题：
降低性能
增加 HTTP 请求
推荐的做法是：
把 Toggle 数据加载到本地 Collection 中。
之后所有 Toggle 判断都从 本地 Collection 读取。

规则五：Every Toggle Needs an Owner（每个 Toggle 必须有 Owner）
每一个 Toggle 都必须有明确的负责人。
如果没有 Owner，就会出现这些问题：
Toggle 被遗忘,没有人负责清理
Toggle 长期存在, 超过一段时间就会变为僵尸toggle。
建议为 Toggle 增加一些元数据，例如：
字段
示例
Owner
Team A
CreatedDate
ExpireDate
这样可以更好地管理 Toggle 生命周期。

规则六：Toggles Are Temporary（Toggle 是临时机制）
Feature Toggle 的本质是 功能发布工具，而不是长期配置。
当功能已经稳定并全面上线之后，Toggle 应该被删除。
一个典型的生命周期是：
复制代码

Create
↓
Rollout
↓
Stable
↓
Remove
如果 Toggle 长期存在，就说明它已经偏离了最初的设计目的。

规则七：Kill the Zombie Toggles（清理僵尸 Toggle）
长期保持开启状态的 Toggle 被称为 Zombie Toggle（僵尸 Toggle）。
Zombie Toggle 会带来很多问题：
不必要的配置复杂度
难以理解的代码路径

因此需要定期进行 Toggle Review，并清理已经不再需要的 Toggle。

第三部分：Real Project Patterns and Play Actions
接下来我介绍几个 在 DCPCC 项目中真实发生的案例，以及我们采取的改进措施。
这些案例可以帮助我们理解为什么需要这些规则。

Pattern 1：长期存在的 Toggle
在项目中发现，有些 Toggle 已经开启超过 3 年，并且没有任何修改记录。
这些 Toggle 已经不再起到 Feature Flag 的作用，而是变成了类似 永久配置开关。
改进措施

我们做了以下工作：
对所有 Toggle 进行 整体review
找出长期存在的 Toggle
删除确认不再需要的 Toggle
为仍然需要的 Toggle 明确 Owner 和用途
目前这项管理工作 仍在持续进行中。

Pattern 2：Toggle Explosion
在项目中我们发现两个 Toggle：
LGD Metrics
Borrow Group Management
这两个 Toggle：
一个有 30 条记录
一个有 25 条记录

这些记录实际上是在控制 同一类功能行为的不同情况。
这说明 Toggle 被当成了一种 复杂配置机制，而不是简单的 Feature Flag。

改进措施
我们正在进行以下优化：
重新review Toggle 数据，
逐步减少冗余 Toggle 数据

Pattern 3：重复查询 Toggle 导致性能问题
在一些页面中，Toggle 是 在使用时直接查询数据源，而没有缓存。
这会导致：
同一个 Toggle 被重复查询，产生大量 HTTP Requests
同时，在app启动时，在页面上有很多toggle被分散查询，导致 APP 加载速度变慢

改进措施
我们进行了如下优化：
在 APP 初始化阶段一次性加载 Toggle
将 Toggle 存入 本地 Collection
后续逻辑统一从 Collection 读取
这样大幅减少了 HTTP 请求数量，并提升了加载性能。

Pattern 4：Toggle 与业务逻辑混在一起
在部分代码中，Toggle 判断 直接嵌入在复杂业务逻辑中。
这样会导致：
代码可读性降低
新开发人员难以理解
维护成本增加
改进措施
我们正在逐步优化：
将 Toggle 判断与业务逻辑分离
统一 Toggle 访问方式
简化 Toggle 条件逻辑
这样可以明显提升代码可维护性。

最后总结一下。
Feature Toggle 是一个非常重要的机制，可以帮助我们：

安全发布功能，降低上线风险
灵活上线，提高系统灵活性
出现功能问题，快速回滚，在DCPCA当中已经得到了多次验证

但如果缺乏管理，它也很容易变成 tech debt。
所以我们呼吁，所有开发人员在使用 Feature Toggle 时，都遵循这七条规则。
这样可以确保 Toggle：
更易管理
更容易理解。

最后，对待toggle，应该把他看做为一种零时工具，而不是永久的架构。
